## 1. 怎么收集有效的反馈

### 不要问「你觉得怎么样？」

<DataTable
  headers={["不好的问题", "好的问题"]}
  rows={[
    ["你觉得怎么样？", "你用这个完成了什么任务？"],
    ["好用吗？", "哪个步骤让你困惑或卡住了？"],
    ["你愿意付费吗？", "如果明天这个产品消失了，你会想念它吗？"],
    ["有什么建议？", "你最想加的一个功能是什么？为什么？"],
  ]}
/>

<KeyPoint emoji="🎯">
  观察用户行为 > 听用户说什么。如果用户说「挺好的」但不用——那不是好信号。如果用户说「这里有点麻烦」但一直在用——那才是真需求。
</KeyPoint>

### 反馈分类

```
💚 正向信号（用户真的需要）:
  - 主动问「什么时候上线正式版？」
  - 用了之后截图分享给别人
  - 反复使用、提具体改进建议

💛 中性信号:
  - 「挺好的，以后可能会用」
  - 提了不切实际的功能需求

❤️ 负向信号（需要立刻改进）:
  - 试了一次就不用了
  - 说「不知道这个能干嘛」
  - 核心功能报错或太慢
```

## 2. 快速迭代

根据反馈，挑 1-2 个最重要的问题修复/改进——不要全部改：

```
迭代优先级:
1. 核心功能 Bug（用户想用但用不了）
2. 新用户上手体验（不知道怎么开始）
3. 性能问题（太慢）
4. 功能增强（nice to have）
```

<TipBox title="迭代原则" icon={Lightbulb}>
  第一版修复导致用户「不能完成任务」的 Bug。第二版优化新用户的「第一次体验」。第三版才考虑加功能。大多数产品在第一版和第二版之间就死了——不是因为功能不够多，是因为核心体验不流畅。
</TipBox>

## 3. 写产品介绍 + 思考定价

### 一句话介绍模板

```
[产品名] 是一个 [做什么的] ，帮助 [目标用户] 在 [什么场景] 下 [解决什么问题] 。

例：DocAI 是一个文档问答工具，帮助研究人员在阅读论文时快速找到关键信息，不用逐页翻 PDF。
```
